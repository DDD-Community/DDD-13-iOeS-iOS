## 산출물

`docs/<TICKET>/ui-test-cases.md` 한 장. 스킬이 Phase A 시작 시점에 빈 셸로 미리 만들어 두었을 것 — 헤더만 있고 본문은 비어 있는 상태.

## 표 컬럼 (8개, 누락 금지)

| case id | 컴포넌트 | 상태/입력 조건 | 테마 | 언어 / DynamicType | Light/Dark | 디바이스 | 기대 시각 결과 | 스냅샷 파일명 |
|---|---|---|---|---|---|---|---|---|

각 컬럼 작성 규칙:

- **case id**: `<screen>-<component>-<variant>` 형태. 예: `spot-detail-header-sunset-light`. 공백 없이 kebab-case
- **컴포넌트**: 화면 프롬프트 §11 Figma 비교 루프 표의 컴포넌트명과 일치
- **상태/입력 조건**: ViewModel `LoadState`나 도메인 입력값. "거리 nil", "북마크 토글된 상태" 등
- **테마**: 도메인 분기가 있는 화면만(노을/윤슬 등). 없으면 `—`
- **언어 / DynamicType**: 기본 `ko_KR / .large`. 접근성 케이스만 `ko_KR / .accessibilityExtraLarge`
- **Light/Dark**: 케이스 단위로 분리. 한 행에 둘 다 묶지 않음
- **디바이스**: 기본 `iPhone 15`. 다른 사이즈 검증할 때만 명시
- **기대 시각 결과**: "헤더 칩이 노을 배경, PM 6:20 라벨이 트랙상 0.79 위치", "거리 라벨 영역 자체가 숨겨짐" 등 리뷰어가 그림 안 보고도 이해할 만큼
- **스냅샷 파일명**: `<case-id>.png` 와 동일하게. Phase C에서 그대로 사용

## 최소 커버리지 (행 누락 자가 점검)

화면이 다루는 모든 컴포넌트에 대해:

- [ ] **상태 4종**: loading / loaded(정상) / empty(있다면) / error — 화면이 가질 수 있는 만큼만, 억지로 만들지 않음
- [ ] **테마 분기**가 있으면 분기마다 1행씩
- [ ] **선택적 필드 분기**: 거리 nil/있음, 사진 0/1/N장 등 의미 있는 분기
- [ ] **Light/Dark**: 동일 시나리오에서 한 쌍씩 (총 행 수 2배)
- [ ] **DynamicType `.accessibilityExtraLarge`**: 텍스트가 큰 비중인 컴포넌트 최소 1행 (전부 다 만들 필요는 없음)

행 수 가늠: 작은 화면 ~15행, 보통 화면 ~30행, KAN-51 같이 컴포넌트 많은 화면 ~50행 전후.

## 케이스 작성 순서 권장

1. 각 컴포넌트의 "정상 loaded" Light 1행씩 먼저
2. 같은 행을 Dark로 복제
3. 테마/선택 필드 분기 추가
4. 상태 분기(loading/empty/error) 추가
5. 접근성(DynamicType) 1~2행 추가

이 순서가 최소 커버리지 누락을 가장 적게 낸다.

## 안티 패턴

- "기본 케이스 1개만 찍고 나머지는 PR 후 추가" — 매트릭스가 불완전한 채 머지되면 회귀 보호 못 함. 처음 한 PR에서 완결시킴
- Light/Dark를 한 행에 묶고 "동일 결과"라고 적기 — 둘은 다른 스냅샷 파일이다. 행 분리
- "예상 결과" 컬럼에 "Figma와 동일" 만 적기 — 리뷰어가 추가 클릭 없이 이해 못 함. 텍스트 기반 묘사 필요
- 접근성 DynamicType을 통째로 생략 — 텍스트 잘림이 가장 흔한 회귀. 최소 1~2행은 둠
- ui-test-cases.md를 PR 마무리에 "정리 차원에서" 작성 — Phase C 스냅샷이 이걸 베이스로 작성되어야 의미 있음. 순서 거꾸로 가면 표는 사후 합리화 문서가 되어버림
